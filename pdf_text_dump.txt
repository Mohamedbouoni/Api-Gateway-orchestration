Abstract
This end-of-study project was carried out at Harvest Group Tunisia as part of the require-
ments for obtaining the Bachelor degree in Software Engineering and Information Systems from
Horizon School of Digital Technologies.
Given the large volume of data processed daily, maintaining data integrity is increasingly
challenging. This project implements an automated system called Argus to ensure data integrity
and reliability through continuous monitoring and alerting within the data pipeline. Argus au-
tomatically detects job failures or delays, compares data across stages to verify consistency,
performs reruns when necessary, and immediately alerts teams to resolve persistent issues.
Keywords: Data Integrity, Automation, Monitoring, Argus.

Glossary
Technical terms used in the context of the project:
•Job Templates : Job templates are collections of configurations defining how a specific
job should execute. They include information such as the cron expression, which specifies
when a job should run.
•Job: A Job is a scheduled process with a specific configuration designed to generate
reports from the retailer data source and download them for further processing.
•JobRun : A Job represents the definition of the process described above, and it can have
multiple executions based on the cron expression defined in its configuration. These
individual executions are referred to as JobRuns in Foreman. Each time the job runs, it
creates a new JobRun.
•Job Run Status : A Job Run status describes the state of the job run. It can be in one of
the following states:
– Future : The initial state before the job run starts.
– Enqueued : Indicates that the scheduled execution time defined by the cron expres-
sion has arrived, and the job run has been enqueued for execution.
– Failed : Means that an issue occurred during the execution.
– Partially Failed : Means that one or more data runs have failed.
– Completed : The execution was successful and all data runs were completed suc-
cessfully.
•DataRun : A Data run is the actual process that makes the requests needed to get the data
back from the source.
•Data Run Status : A Data Run status describes the state of the data run. It can be in one
of the following states:
– New : The initial state after the creation or queuing of the job run.
– No Data Found : Means that the uploaded file contains no data.

•Modularity : Modules maintain high internal cohesion and low external coupling. Com-
munication between modules is done using well-defined interfaces, preferably asynchronously.
•Unified database : All modules share a single database schema.
•Monolith Deployment : The application is deployed as a single unit, although modules
can be distributed across multiple hosts.
•Unified Application Process : The entire application runs as one unified process, scaling
on demand.

6.3.5 Key Vault and Secrets Management
Since both of our container apps are using the same functionalities that require configurations

variables poses challenges in consistency and security, especially across multiple containers.
To overcome these limitations, we leverage Azure Key Vault, securely centralizing storage of
sensitive configuration data. Container Apps then access these secrets at runtime using managed
identities, significantly reducing management complexity and enhancing overall security.
