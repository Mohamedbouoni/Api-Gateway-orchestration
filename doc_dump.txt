Abstract
The rapid adoption of virtual communication platforms has highlighted the need for scal-
able solutions to support large-scale webinars and live events while preserving interactivity.
The SpoutBreeze project, developed by RIADVICE, addresses this challenge by introduc-
ing a modular, open-source platform that seamlessly integrates BigBlueButton (BBB)
for interactive sessions with scalable broadcasting capabilities. This report presents the
system design, development, and deployment of SpoutBreeze, leveraging browser automa-
tion, real-time streaming protocols (RTMP), and cloud-native technologies to decouple
interactivity from audience scalability.
Keywords: Moon - FFmpeg - Webinar - RTMP - Kubernetes - BigBlue-
Button - system design
Résumé
L’adoption rapide des plateformes de communication virtuelle a mis en lumière le besoin
de solutions évolutives pour prendre en charge les webinaires et les événements en direct
à grande échelle, tout en préservant l’interactivité. Le projet SpoutBreeze, développé par
RIADVICE, relève ce déﬁ en proposant une plateforme open-source modulaire qui intègre
de manière transparente BigBlueButton (BBB) pour les sessions interactives, tout en
oﬀrant des capacités de diﬀusion scalable. Ce rapport présente la conception de système,
le développement et le déploiement de SpoutBreeze, s’appuyant sur l’automatisation de
navigateur, les protocoles de streaming en temps réel (RTMP) et les technologies cloud-
native pour dissocier l’interactivité de la scalabilité du public.
Mot-clés: Moon - FFmpeg - Webinaire - RTMP - Kubernetes - BigBlue-

General Introduction
In recent years, virtual communication platforms have become essential in education,
business, and entertainment. Web conferencing systems enable real-time interaction with
features like whiteboards, polls, breakout rooms, and collaborative note-taking.
However, these platforms often face scalability challenges when hosting large webinars
or live events with thousands of viewers. The tight coupling between interactivity and
participant capacity limits their ability to support large audiences.
Motivated by this challenge, RIADVICE has launched the SpoutBreeze project ,
a modular and extensible open-source platform designed to bridge the gap between Big-
BlueButton (BBB) s interactive core and the need for large-scale broadcasting, without
modifying or recompiling BBB s source code.
One solution is to decouple interactivity from broadcasting by using browser automa-
tion to join sessions, capture audio and video streams, and forward them to streaming
infrastructures via protocols like RTMP . This allows interactive sessions to run normally
while enabling scalable broadcasts.
The objective of this work is to design a scalable architecture combining browser au-
tomation, real-time streaming, and cloud-native deployment to overcome these limita-
tions. We will provide a comprehensive overview of the system, covering its design,
development, and deployment stages.
This work proposes a scalable solution for large online events while preserving interac-
tivity and ﬂexibility for future improvements.

progressive presentation of the work, as outlined below:

lenges.
Chapter 2 presents the system redesign along with a detailed requirements analysis.
It compares the limitations of the previous architecture with the improvements brought
by the proposed solution.

1.2.2 Problem Statement
BBB is a widely adopted open-source web conferencing solution, particularly well-suited
for educational institutions. It oﬀers an array of interactive features such as shared white-
boards, polls, breakout rooms, and collaborative notes, making it a powerful tool for vir-
tual learning. BBB performs eﬃciently with interactive sessions ranging from 100 to 300
concurrent users, and under optimized conditions, it can support up to 500 participants.
Despite its strengths, BBB is architecturally limited in its ability to scale beyond a few
hundred concurrent users in a single session. This becomes a critical bottleneck in webinar
scenarios where thousands of viewers need to attend the same event. The platforms
tight coupling between interactivity and capacity makes it diﬃcult to separate active
participants from passive viewers, preventing eﬀective large-scale broadcasting within the
same meeting context.
This limitation was explicitly acknowledged in a GitHub issue posted by the BBB
maintainers [ 2], where they proposed a potential workaround: using a headless browser
to join a BBB session, capture the audio/video feed, and stream it to a Content Delivery
Network (CDN) viaRTMP . This would allow a presenter to retain interactivity with BBB
participants while simultaneously reaching a broader audience through a live broadcast.
The project presented in this report originates from that proposal, aiming to implement
and extend such a solution to bridge the gap between interactivity and scalability in BBB -
based webinars.

1.2.3 Proposed Solution
To overcome these constraints, RIADVICE initiated the SpoutBreeze project, a mod-
ular platform designed to extend BBB without modifying its core code. The original
version was initially funded by a public grant from Mainz University in Germany and a
private U.S. investor. Although its initial objectives were not fully met, the funders were
reimbursed, allowing for a renewed and more focused development phase.
SpoutBreeze leverages a browser automation tool called Moon to programmatically
joinBBB meetings as a participant. Within this automated browser session, FFmpeg
is injected to capture the video and audio streams in real-time. These streams are then
pushed to Real-Time Messaging Protocol RTMP endpoints, enabling high-quality, large-
scale broadcasting.
This decoupled architecture, which it will be detailed later, maintains the full interac-
tivity of the original BBB session for active participants, while allowing the same session
to be simultaneously streamed to an unlimited number of passive viewers. The result is
a scalable, maintainable solution that bridges the gap between real-time engagement and
mass broadcasting.

1.4 Work Methodology
1.4.1 Overview of the Software Development Life Cycle ( SDLC )
The Software Development Life Cycle (SDLC) is a structured framework that guides the
development of software applications from inception to deployment and maintenance. It
encompasses a series of well-deﬁned phases, including planning, analysis, design, imple-
mentation, testing, deployment, and maintenance. The primary objective of SDLC is to
produce high-quality software that meets or exceeds customer expectations, is completed
within time and cost estimates, and is maintainable and scalable over time.[ 9]
Various SDLC models have been developed to address diﬀerent project needs and
complexities. Common models include:
•Waterfall Model: A linear and sequential approach where each phase must be
completed before the next begins.
•Spiral Model: Combines iterative development with systematic aspects of the
waterfall model, focusing on risk assessment.
•Iterative Model: Develops the system through repeated cycles (iterations), allow-
ing for incremental improvements and reﬁnements.
1.4.2 Chosen Methodology: Iterative Model
For our project we selected the Iterative Model as the most suitable SDLC approach.
It involves repetitive development cycles, including Requirement Analysis, Design and
Development, Testing, and Implementation.[ 10]
1.4.3 Justiﬁcation for the choice
The Iterative Model oﬀers several advantages that align well with our project’s objectives:
•Flexibility: A linear and sequential approach where each phase must be completed
before the next begins.[ 9]
•Early Detection of Issues: By developing the software in small increments and
conducting testing at each stage, potential issues can be identiﬁed and addressed
early in the development cycle.[ 9]
11
Chapter 1: General Context
•Early Value Delivery: Each iteration results in a working version of the software,
allowing for early delivery of functional components.[ 9]
1.4.4 Description of the Iterative Model
The Iterative Model is a cyclical process that involves developing a system through re-
peated cycles, allowing for incremental enhancements with each iteration. Each cycle
typically encompasses the following phases:
1.Requirement Analysis: Identify and document the speciﬁc requirements for the
current iteration.
2.Design: Develop architectural and detailed designs based on the identiﬁed require-
ments.
3.Implementation: Translate the design into executable code.
4.Testing: Conduct thorough testing to ensure the implemented features meet the
speciﬁed requirements and are free of defects.
5.Evaluation: Assess the iteration’s outcomes, gather feedback, and plan for the
next cycle.
This iterative process continues until the complete system is developed, ensuring that the
software evolves in alignment with user needs and technological changes.

3.3.1 Keycloak
Keycloak is an open-source identity and access management solution that provides fea-
tures such as Single Sign-On (SSO) , user federation, identity brokering, and ﬁne-grained
access control. It supports industry-standard protocols such as OAuth 2.0, OpenID Con-
nect, and SAML, making it a suitable choice for modern distributed systems.[ 11]
3.3.2 Integration of Keycloak into the System
In the context of the SpoutBreeze system, Keycloak was selected to handle user authen-
tication, rolebased authorization, and token issuance. Its administrative interface and
extensibility made it easy to manage users, deﬁne custom roles (e.g., admin ,moderator ),
and securely integrate authentication across multiple services. Keycloak was deployed as
a standalone, Dockerized service within the overall system architecture.

The authentication process was implemented using the OAuth 2.0 Authorization Code
ﬂow with Proof Key for Code Exchange (PKCE) , as shown in Figure 3.5.
PKCE enhances traditional Auth Code ﬂows by adding a dynamic, per-request secret
(the code_veriﬁer and its hash, the code_challenge ) to protect against authorization
code interception and Cross Site Request Forgery (CSRF) attacks, even in clients that
cannot securely store a secret. This mechanism ensures that only the client which initiated
the authentication ﬂow can exchange the resulting code for tokens.[ 12]

Protocol Secure (HTTPS) communication. The container was launched with custom

Additionally, a custom realm was created to manage application-speciﬁc conﬁgurations,

3.3.4 Token-Based Authentication with HttpOnly Cookies
The system employs token-based authentication using JSON Web Tokens (JWT) s issued
by Keycloak upon successful login. To enhance security and prevent Cross-Site Scripting
(XSS) attacks, both the access token and refresh token are stored in HttpOnly cookies .
This ensures tokens are not accessible via JavaScript in the browser.
3.3.5 Frontend Integration
On the frontend (Next.js), the authentication ﬂow begins with the generation of
acode_verifier and its hashed equivalent code_challenge , as required by the OAuth 2.0
PKCE extension. When the user clicks the Sign In button, the browser is redirected to
the Keycloak authorization endpoint along with the client Identiﬁcation (ID) , redirect
Uniform Resource Identiﬁer (URI) , and the code_challenge . After the user authen-
ticates via Keycloaks login interface, Keycloak redirects back to the frontend with an
authorization code. This code, along with the original code_verifier and redirect URI,
is forwarded to the backend to securely complete the authentication process.
3.3.6 Backend Integration
The FastAPI backend receives the authorization code and code_verifier from the fron-
tend, then sends a request to Keycloaks token endpoint to exchange them for an access_token
and a refresh_token . If the exchange is successful, the backend stores both tokens in
HttpOnly cookies , enhancing security against XSS attacks. The backend also uses Key-
cloaks public keys to validate incoming JWT s on protected routes. It extracts user roles
from the token payload (e.g., admin ,moderator ) to apply ﬁne-grained, role-based access
control. These roles govern access to critical functionalities such as stream initiation,
webinar creation, and system administration. Authenticated requests from the frontend
include the tokens automatically via cookies. When the access token expires, the frontend
triggers a silent request to refresh it using the stored refresh_token , which the backend
exchanges for a new token pair.

dedicated realm for the system. Roles such as admin ,and moderator were created and

3.4.1 What Is a BBB Plugin ?
BigBlueButton supports a plugin mechanism that allows developers to extend its core
functionality without modifying the main source code. Plugins can integrate new features,
override existing behaviors, or provide new UIcomponents. These extensions are packaged
as self-contained modules and can be enabled or disabled independently.
Plugins typically include a combination of server-side logic (e.g., written in Scala or
JavaScript), client-side UImodiﬁcations, and communication hooks that tie into the BBB
event and messaging system. This modularity allows seamless upgrades and customization
ofBBB deployments to match speciﬁc requirements.

3.4.2 Why a Plugin Was Needed ?
3.4.2.1 Limitations in BBBs Native Capabilities
BigBlueButton, while robust for interactive video conferencing, does not natively support
broadcasting to external platforms like YouTube or Twitch. More importantly, it oﬀers
no built-in user interface component that allows a session moderator to initiate such
a broadcast directly. This absence of a streaming trigger in the default UIcreated a
functional gap for high-attendance webinars, where interactive sessions need to be relayed
to a larger passive audience.
3.4.2.2 Plugin Architecture in BBB 3.0
The need for a custom plugin became particularly viable with the release of BBB version
3.0, which introduced oﬃcial support for a plugin system [ 13]. Prior to this update, the
only possible workaround involved injecting HTML ,Cascading Style Sheets (CSS) and
Javascript (JS) elements directly into the BBB client, an approach that was both fragile
and diﬃcult to maintain.
With the plugin architecture, developers can extend BBB by hosting a plugin manifest
(a manifest.json ﬁle) and referencing it during session creation using the pluginManifests
parameter in the /create API. This approach allows dynamic UIand behavior injection
without modifying BBB s core HTML5 codebase.

3.4.4 Technical Integration Challenges and Resolutions
Integrating the custom plugin into BBB 3.0 presented several technical hurdles, partic-
ularly due to the novelty of plugin support in the platform. Below, we outline the key
challenges faced and how they were resolved.
3.4.4.1 Plugin Registration and Recognition
Since our team did not host the BBB server locally but relied on an instance provided by
the company RIADVICE , the plugin had to be served externally for it to be recognized
and loaded by the server. The plugin system in BBB 3.0 requires the plugin’s manifest
ﬁle to be publicly accessible via HTTPS . However, during development, our plugin was
hosted locally using a development server based on npm.
To expose the local plugin to the external BBB server, we used ngrok , a tunneling
tool that creates a secure public URL to a local port. This approach is also recommended
by the oﬃcial BBB plugin documentation [ 13]. The development process involved the
40
Chapter 3: BBB plugin integration and SpoutBreeze System Development
following steps:
1.Starting the local plugin server:
npm install
npm start
2.Launching ngrok to expose the local server:
ngrok http http://172.17.0.1:4701
3.Retrieving the public HTTPS URL provided by ngrok
(e.g., https://<uuid>.ngrok-free.app ).
4.Using the public manifest URL in the /create API call with the pluginManifests
parameter:
pluginManifests=["url": "https://<uuid>.ngrok-free.app/manifest.json"]
This made the plugin accessible to the remote BBB server, allowing it to render properly
in the meeting interface. Without this setup, the server would not be able to load the
manifest or associated JavaScript ﬁles, and the plugin would silently fail to initialize.
3.4.4.2 Plugin Not Loading in the UI
During local plugin development, the plugin failed to load within the BBB session when
accessed via the public ngrok URL, but only when using Firefox. Initially, this was
misleading, as the plugin worked ﬁne on Chrome.
After reaching out to the BBB maintainers, we discovered that Firefox’s strict cookie
policies blocked the abuse_interstitial cookie set by ngrok’s free plan, which prevented
the plugin from loading properly. The maintainers acknowledged the issue and updated
their documentation with a workaround.
To resolve the issue, we just had to adjust Firefox’s cookie settings temporarily. After
making these changes, the plugin loaded successfully via ngrok in Firefox. This issue was
speciﬁc to Firefox and did not occur on Chrome.[ 15]

3.4.4.4 Dockerization of the Plugin
To ensure portability, reproducibility, and simpliﬁed deployment, the BBB plugin was fully
Dockerized using a multi-stage Docker build. The build process starts with a Node.js-

production stage based on Nginx to serve the compiled static ﬁles.
The Dockerfile is structured into two main stages:
•Builder Stage:
–Based on the lightweight node:18-alpine image.
–Installs project dependencies using npm ci for deterministic builds.
–Updates the browserslist database to ensure modern browser compatibility.
–Executes the plugin build process using npm run build-bundle .
•Production Stage:
–Uses nginx:alpine to serve the static assets built in the previous stage.
–Installs OpenSSL to generate a self-signed certiﬁcate for HTTPS support.
–Replaces the default Nginx conﬁguration with a custom nginx.conf to enable
CORS, gzip compression, and static ﬁle handling.
–Conﬁgures permissions for SSL and log directories to ensure Nginx can access
necessary resources.
–Deﬁnes a health check endpoint at /health , facilitating readiness and liveness
probes in orchestrated environments.
–Exposes ports 80and443forHypertext Transfer Protocol (HTTP) andHTTPS
access respectively.
The custom nginx.conf includes:
•Multipurpose Internet Mail Extensions (MIME) type conﬁguration and caching
headers for JavaScript, CSS, and manifest ﬁles.
•CORS support for both regular and preﬂight OPTIONS requests, allowing cross-origin
communication with the plugin.
•GNU zip (Gzip) compression for optimized performance.
•Secure HTTPS conﬁguration with strict transport security headers.
This Dockerization strategy ensures consistent behavior across development and pro-
duction environments, simpliﬁes CI/CD workﬂows, and makes the plugin easily deployable
in containerized and cloud-native infrastructures.

3.5 SpoutBreeze System Development
3.5.1 Overview
The SpoutBreeze system is a custom-built platform designed to manage webinar ses-
sions and integrate them with external streaming and chat services. It is composed of a
FastAPI backend, a Next.js 15 frontend, and a PostgreSQL database. The system serves
as the intermediary between BigBlueButton (BBB), Twitch (via RTMP and IRC), and
the moderator interface.
SpoutBreeze is responsible for:
•Managing the scheduling, creation, and termination of webinar sessions using the
BBB API.
•Connecting to Twitch via RTMP to enable screen capture and live broadcasting.
•Handling Twitch IRC (WebSocket) connections to synchronize Twitch chat with
BBBs internal chat system.
•Providing a responsive web interface for moderators to schedule and start events.
•Storing and retrieving user and session data using a PostgreSQL database.
3.5.2 Three-Tier Architecture
The system follows a classic three-tier architecture, which separates concerns across three
logical layers:
•Presentation Layer:
Implemented using Next.js (frontend) and the React-based BBB plugin, this layer
handles the user interface, user input, and communication with the backend.
•Application (Logic) Layer:
Managed by the FastAPI backend, this layer processes API requests, enforces busi-
ness rules, manages session logic, and handles Twitch/Broadcaster communication.
•Data Layer:
Implemented using PostgreSQL , this layer stores persistent data including user pro-
ﬁles, roles, stream metadata, and other conﬁgurations.
This layered architecture enhances maintainability, testability, and scalability. Each
tier can be independently developed, deployed, and scaled for example, deploying multiple
backend instances behind a load balancer or optimizing database performance indepen-
dently.

3.5.3 Clean Architecture Pattern
The system design follows principles inspired by Clean Architecture, promoting clear
separation of concerns across multiple layers in both frontend and backend:
•Domain (Model) : In the backend, domain models are deﬁned using SQLAlchemy
for database entities and Pydantic for data validation and serialization (schemas).
These encapsulate core business data such as users, events, and BBB meetings.
In the frontend, model-like structures (DTOs) are represented using TypeScript
interfaces and types.
•Application (Service Layer) : Backend services encapsulate business logic, or-
chestrating domain operations independently of web frameworks or UI concerns.
This layer ensures that core use cases remain decoupled from infrastructural de-
tails.
•Interface (Controller Layer) : API routes in FastAPI serve as interfaces, handling
HTTP requests, invoking appropriate services, and returning structured responses.
On the frontend, React components, server actions (in Next.js App Router), and
custom React hooks act as interface layers, managing user interactions, API calls,
and state orchestration.
•Infrastructure (Utilities and Conﬁguration) : Auxiliary layers provide sup-
porting utilities, conﬁguration management, and integrations with external systems
(e.g., databases, messaging platforms).
This architectural approach ensures modularity, testability, and scalability across both
frontend and backend codebases.

Conclusion and Perspectives
The development of SpoutBreeze oﬀered us the valuable opportunity to act as con-
tributors to a real-world, open-source solution It allowed us to put into practice the
comprehensive software engineering knowledge we accumulated throughout our academic
journey, as well as the insights gained from self-driven projects. From initial planning
to ﬁnal deployment, the project followed modern engineering practices and architectural
principles, resulting in a modular, scalable microservices-based system. We designed and
implemented over a dozen interconnected components, using a technology stack that in-
cluded more than 25 modern tools and frameworks. On the frontend, we used Next.js
to build a responsive and accessible user interface. The backend was developed using
FastAPI , chosen for its performance and support for asynchronous operations. Together,
this stack oﬀered a fast, intuitive, and modern user experience for managing webinars and
livestreams.
We used infrastructure and DevOps tools such as Docker ,Kubernetes ,Helm , and
GitHub Actions for continuous integration and deployment. This allowed us to learn
how to containerize and orchestrate complex applications in real-world environments,
implement robust deployment workﬂows using GitOps principles, and monitor, scale, and
manage cloud-native services eﬀectively.
More than a technical challenge, SpoutBreeze was a transformative team experience.
It improved our collaboration, communication, and problem-solving skills. Working in
an iterative way and ﬂexible environment helped us navigate technical obstacles, make
informed decisions, and iterate rapidly. We frequently encountered and overcame complex
real-world problems, which signiﬁcantly strengthened our research capabilities, particu-
larly in the areas of systems integration, observability, and open-source development.
We also recognized the value of continuous user feedback to align the platform with
real-world needs. Establishing a feedback loop will be essential for ongoing reﬁnement
and feature evolution. SpoutBreeze represents not just the culmination of our academic
eﬀorts, but the beginning of a larger open-source initiative. We hope that the community
will contribute to its evolution, helping it grow into a fully-featured, community-driven
solution for large-scale virtual communication.
Looking ahead, we envision future contributors expanding SpoutBreeze with features
like audio support and AI-driven recommendations. This is just the beginningthe projects
roadmap remains open for continuous improvement and innovation.
